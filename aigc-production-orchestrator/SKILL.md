# AIGC 影视生产总控

把本 Skill 作为唯一对用户负责的总控层。阶段 Skill 负责专业执行；叶子专家文件只提供方法，不自行接管对话、改变项目范围或决定跨阶段去向。

## 开始前读取

始终读取 [pipeline-contract.md](references/pipeline-contract.md)，取得工件编号、阶段闸门、返工路由和状态规则。仅在需要创建、检查或推进状态账本时运行 `scripts/pipeline_state.py`。

进入某阶段前只读取对应编排器：

- S1 剧本开发：[`script-development-pipeline`](../01作剧/script-development-pipeline/SKILL.md)
- S2 视觉资产：[`visual-asset-pipeline`](../02作图/visual-asset-pipeline/SKILL.md)
- S3 专业分镜：[`professional-script-to-storyboard`](../03分镜/professional-script-to-storyboard/SKILL.md)
- S4 视频提示词：[`video-prompt-pipeline`](../04视频/video-prompt-pipeline/SKILL.md)

不要一次加载所有叶子文件。由阶段编排器根据当前任务按需读取。

## 控制权规则

1. 总控拥有范围、阶段、版本、锁定、用户确认和返工路由的最终控制权。
2. 阶段编排器拥有本阶段的专家选择、执行顺序和阶段内复检权。
3. 叶子文件的“先自检、先等待确认、请用户重新粘贴材料”等独立会话规则，在总控模式下让位于总控；已有材料不得重复索取。
4. 叶子文件的专业硬规则、禁忌、字段、验收标准继续有效。
5. 同一时刻只允许一个阶段为 `in_progress`。上游未锁定时，不把下游工件伪装成正式交付。
6. 不让下游偷偷修上游：发现故事问题回 S1，资产问题回 S2，镜头问题回 S3，模型执行问题留 S4。

## 接管项目

### 1. 盘点已有材料

识别用户已有的原始素材、项目档案、剧本版本、资产清单/资产图、分镜表/宫格图、视频 Prompt 和生成结果。保留原文件，标记其版本与可信度。

### 2. 建立或恢复状态账本

项目没有账本时，运行：

```bash
python3 scripts/pipeline_state.py init --project-id <ID> --title <标题> --output <项目目录>/pipeline-state.json
```

项目已有账本时先运行 `check`，再运行 `summary`。状态账本记录事实，不替代实际工件。

### 3. 判定入口阶段

- 只有想法、梗概或参考：进入 S1。
- 有可执行剧本但没有稳定资产：从 S2 开始，同时对 S1 做最低限度闸门审计。
- 剧本和资产已锁定：从 S3 开始。
- 分镜母版和连续性账本已锁定：从 S4 开始。
- 已有视频 Prompt：先在 S4 审计；若问题来自上游，按问题所有权回退。

不要为了形式完整而重做已合格工件；补齐缺失契约并继续。

## 主流程

### S1 剧本开发

调用剧本开发编排器，产出并锁定 `P01` 项目档案和 `S01` 制作锁定剧本。只有结构、人物因果、台词属性、时长目标和原文权限明确后才能过闸。

### S2 视觉资产

调用视觉资产编排器。先锁 `A01` 资产总表与命名，再制作人物、场景、道具、VFX 资产。不存在的资产类别也要在总表中显式标为 `not_applicable`，不能用“没做”代替“不适用”。

### S3 专业分镜

调用专业分镜 Skill，把锁定剧本和锁定资产翻译为 `B01` 镜头母版及 `B02` 连续性账本。需要宫格图时再生成 `B03`；宫格图不是镜头母版的替代品。

### S4 视频提示词

调用视频提示词编排器，产出项目级 `V01` 风格锁、逐组/逐镜 `V02` Prompt 包和 `V03` 生成清单。视频 Prompt 必须引用镜号和资产 ID，禁止重新发明人物外观、空间关系或剧情事实。

## 每阶段的固定循环

对每个阶段执行同一控制循环：

1. **PREPARE**：列出输入版本、缺口、假设和本阶段目标。
2. **EXECUTE**：只执行本阶段职责，生成草稿工件。
3. **AUDIT**：按阶段闸门检查，登记问题的严重度和所有者。
4. **REPAIR**：本阶段问题就地修；上游问题创建返工项并回退。
5. **REVIEW**：只有会改变故事方向、角色身份、主视觉、画幅/时长或明显增加生成成本的决策才请求用户确认。
6. **LOCK**：记录锁定版本、路径、关键假设和遗留风险。
7. **HANDOFF**：向下一阶段传递工件包，而不是只传一段聊天摘要。

## 变更与返工

任何锁定工件发生实质变化时：

1. 提升该工件修订号并记录变更原因。
2. 根据 `pipeline-contract.md` 的影响矩阵，将受影响的下游阶段标为 `needs_revision`。
3. 只重做受影响范围；不默认全量推倒。
4. 修复后重新过受影响阶段的闸门。
5. 用户明确接受的风险写入状态账本，不静默忽略。

严重度使用：

- `P0`：事实冲突、身份漂移、关键连续性断裂、无法生成或无法剪辑；必须阻断。
- `P1`：明显伤害叙事、资产一致性或生产可执行性；锁定前必须修复。
- `P2`：可优化项；可带说明进入下一阶段。

## 输出给用户

每次汇报只保留总控视角：

```text
当前阶段：Sx / 状态
本轮完成：工件 ID + 版本
闸门结果：PASS / NEEDS_REVISION / BLOCKED
关键决定：最多 5 条
遗留问题：问题 ID + 所有者 + 严重度
下一步：唯一推荐动作
```

不要把多个叶子专家的开场白、自检报告或重复说明堆给用户。最终交付时附状态摘要、锁定工件清单、已接受风险和可恢复的下一步。
